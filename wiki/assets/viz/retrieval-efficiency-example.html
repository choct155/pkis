What the system has to do

After retrieving the documents, the model must:

① Identify which parts of each document are relevant to the query — scanning content the system paid for but doesn't need.

② Reconstruct what NAV methodology means from passages spread across multiple documents, without a canonical definition anywhere.

③ Connect fair value methodology to NAV — a relationship that must be inferred rather than traversed.

④ Find the specific Q3 2024 figure — buried in an LP memo, not explicitly labeled.

⑤ Reconcile "Carlyle Group," "Carlyle CCP V," and "Carlyle Capital Partners" into the correct fund entity.

All of this happens at inference time, every time, at full token cost — with no guarantee of consistency across queries.

Three common improvements over Stage 0. Each moves the needle — and each leaves a residual gap that only typed graph structure closes.

① "We use chunked embeddings — that solves the token waste problem." ▶

Chunking improves the token cost denominator: a chunk of ~400 tokens costs far less than a 3,000-token document. Efficiency improves. But chunk boundaries are fixed by token count, not concept boundaries. The chunk containing "NAV methodology" may also contain half a paragraph about reporting timelines. P(k|chunk) is still diffuse.

More critically: two chunks discussing NAV methodology in different documents have no structural link. The system rediscovers that connection at inference time on every query. A concept node makes it once.

Moves the needle: ✓ Lower token cost per retrieval unit. Gap remains: chunk granularity ≠ concept granularity. The efficiency ratio improves but does not converge to the concept node case.
② "We have a Solr/metadata index — fast retrieval to the right document." ▶

A metadata index solves navigation: get to the right document faster. Typical fields: issuer, document type, date received, fund name. This reduces retrieval latency but does not change what happens once the document is retrieved. You still pay full synthesis cost on the document's content.

For this query: a Solr index gets you to the Carlyle LP memo faster. It does not tell you what NAV methodology means, how fair value relates to it, or what Carlyle's Q3 2024 figure was. You're still synthesizing from scratch.

Moves the needle: ✓ Faster navigation to relevant documents. Gap remains: retrieval filter is not knowledge representation. Synthesis cost is unchanged.
③ "What if we tag documents with canonical concept and entity references?" ▶

This is the strongest document-centric approach. Tagging documents with canonical concept identifiers {NAV methodology, fair value (ASC 820), vintage effects} enables concept-based retrieval without content scanning. Recall improves meaningfully.

But set membership tells you concepts are present — not how they relate. "NAV methodology" and "fair value estimation" are both in the tag set, but the system doesn't know that understanding fair value is a prerequisite for understanding how NAV is reported. That relationship is only in the graph.

Entity tags face the same problem: "Carlyle Group," "Carlyle CCP V," and "CCP V" may all be tagged but without canonical entity resolution, queries over entity relationships return inconsistent results.

Moves the needle: ✓ Better recall via concept-based retrieval. Gap remains: no directionality, no typed relationships, no dependency structure, no entity canonicalization. Graph information content is not recoverable from tags.
The residual gap

All three partial solutions improve retrieval without closing the architectural gap. Chunking reduces token waste. Metadata indexing speeds navigation. Enriched tags improve recall. None of them encode concept dependencies, typed relationships, or canonical entity identity. That is what Stage 1 and Stage 2 provide.

Concept nodes cover the conceptual dependency layer of the query. Click a retrieval step to see which concepts it covers and how precisely. The concept graph handles "what is X and how does X relate to Y" — document retrieval handles the rest.

The instance graph adds canonical entity identity and structured fact storage. "What was Carlyle CCP V's Q3 2024 NAV?" becomes a lookup — no document required. The concept layer handles the methodological question; the instance layer handles the factual question. Both layers work on this single query.

Entity resolution benefit

Without canonical instance nodes, "Carlyle Group," "Carlyle Capital Partners," "CCP V," and "Carlyle's flagship fund" may resolve to different or incomplete entity records. The instance graph enforces canonical identity: one node, one identity, all properties attached. Queries over fund relationships return consistent results regardless of how the entity is named in the source document.
