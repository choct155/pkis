Formal Derivation · Retrieval-Inference Tradeoff Cluster

Coverage-Per-Token Efficiency:
Concept Graph vs. Document Retrieval

PKIS Research Program v0.1 · June 2026 Status: staged hypothesis
Primitives

We establish the formal objects required for the derivation. All subsequent results follow from these definitions.

q A natural language query posed to the system.
𝒞 The universe of all concepts — both those encoded in the graph and those recoverable only from world knowledge.
C(q) A probability distribution over 𝒞 induced by q. P(c | q) is the weight of concept c in answering q — the fraction of q's information need attributable to c. Formally: Σc ∈ 𝒞 P(c | q) = 1.
G A multipartite knowledge graph whose nodes include typed concept nodes, source nodes, and relational edges.
n A retrieved node — either a concept node c ∈ G or a document node d ∈ G.
tokens(n) The token cost of consuming node n — proportional to its length in the context window.
I(q ; n) The mutual information between query q and node n: the expected reduction in uncertainty about C(q) obtained by retrieving n.
H(q | S) Residual entropy about q given retrieved set S — how much of C(q) remains uncovered.
ε Sufficiency threshold: retrieval terminates when marginal information gain ΔI(q ; n | S) < ε.
Kc Graph construction cost per concept node — token cost of encoding a concept node at integration time.
N Query volume — number of queries over the amortization horizon.
Note on C(q) C(q) is not directly observable. In practice it is approximated via semantic bridging — mapping query tokens to candidate concept nodes by embedding-space distance. This approximation problem is treated as separable from the coverage computation and deferred. The derivation below conditions on a well-estimated C(q).
Coverage of a Query by a Node Set

Define the coverage of query q by graph G as the expected mass of C(q) that falls on encoded nodes:

Definition — Coverage
Coverage(q, G) = Σc ∈ G P(c | q) · 𝟙[c ∈ G]

// Each concept node is a Bernoulli trial against C(q).
// Mass on un-encoded concepts must be recovered at inference time — at full model cost.

Retrieval proceeds greedily by marginal information gain. Given a retrieved set S, the marginal gain from adding node n is:

Marginal Information Gain
ΔI(q ; n | S) = I(q ; n, S) − I(q ; S)

// Add nodes in descending order of ΔI until ΔI < ε.
// ε is the sufficiency threshold — a design parameter.

The retrieved set S* at termination satisfies H(q | S*) < ε. Any mass of C(q) not covered by S* is handled at synthesis time — either recovered from model world knowledge or lost.

Document Retrieval: The Token Waste Problem

A document d is a multi-concept artifact. Its information content spans many concepts simultaneously:

Document information structure
P(k | d) = distribution over knowledge units contained in d

// P(k | d) is broad: most knowledge units in d are irrelevant to q.
// But tokens(d) is paid in full regardless of relevance.

The mutual information between q and document d decomposes as:

Document mutual information
I(q ; d) = I(q ; d)relevant + I(q ; d)irrelevant

// Only the relevant portion contributes to Coverage(q, G).
// The irrelevant portion costs tokens but yields zero mutual information with q.

Define the relevance ratio of document d with respect to query q:

Relevance ratio
ρ(q, d) = I(q ; d)relevant / tokens(d)

// ρ is structurally low for documents: purpose-built artifacts with broad scope.
// ρ → 0 as document length increases and query scope narrows.
Structural failure mode Even when a document contains the relevant concept, the token cost is paid at document scale, not concept scale. The coverage obtained does not scale with token expenditure. This is not a model quality problem — it is an architectural mismatch between document granularity and query granularity.
Concept Graph Retrieval: Efficiency by Construction

A concept node c is scoped to a single knowledge unit by design. Its information structure is narrow:

Concept node information structure
P(k | c) ≈ δk=c

// Approximately a point mass on the concept it encodes.
// I(q ; c) is high when c ∈ support(C(q)), near zero otherwise.

Define the coverage-per-token efficiency for a retrieved node n:

Definition — Retrieval Efficiency
Eff(q, n) = I(q ; n) / tokens(n)

// Mutual information gain per token consumed.

The architectural claim follows directly from the information structures of c and d:

Efficiency dominance
𝔼[Eff(q, c)] > 𝔼[Eff(q, d)]

// Because: I(q ; c) ≈ I(q ; d|relevant) — comparable signal
// But: tokens(c) << tokens(d) — far lower cost
// The gap widens as doc length ↑ and query scope ↓.
Amortization: Lower Bound on Cumulative Savings

Graph construction is a one-time cost per concept node. Query-time retrieval is a per-query cost. The amortization result shows the crossover at which accumulated efficiency savings recover construction cost.

Let a query q require coverage of k concept nodes from G. Define the per-query token costs:

Per-query token cost
Costdoc(q) = k · Td // k documents retrieved, each at full doc length
Costgraph(q) = k · Tc // k concept nodes retrieved, each scoped

// T_d = avg document tokens. T_c = avg concept node tokens. T_d >> T_c.

The per-query savings from graph retrieval versus document retrieval:

Per-query savings
Δ(q) = k · (Td − Tc)

Cumulative savings across N queries, against total construction cost for k concept nodes:

Cumulative savings vs. construction cost
Savings(N) = N · k · (Td − Tc) − k · Kc

// Construction cost: k nodes × K_c tokens each (paid once).
// Crossover at N* = K_c / (T_d − T_c).
Crossover query volume
N* = Kc / (Td − Tc)

// For N > N*: graph retrieval is strictly cheaper in cumulative terms.
// This is a LOWER BOUND — does not yet include dependency chain savings.
Lower bound interpretation This calculation assumes no dependency chain benefit — concept nodes are retrieved independently with no graph traversal. The actual savings are strictly higher once pre-encoded concept dependencies are accounted for (see open extension). The crossover N* is therefore an upper bound on when the architecture pays off.

Proof Sketch: Efficiency Dominance
(i)

A concept node c encodes a single knowledge unit by construction. Therefore tokens(c) = Tc where Tc is bounded and small — proportional to the depth of a single concept explanation.

(ii)

When c ∈ support(C(q)), I(q ; c) ≈ P(c | q) · H(c) — the full information content of concept c, weighted by its contribution to answering q. When c ∉ support(C(q)), I(q ; c) ≈ 0.

(iii)

A document d contains many knowledge units. Its token cost Td is paid in full regardless of which knowledge units are relevant to q. The relevant fraction ρ(q, d) = I(q ; d)relevant / tokens(d) satisfies ρ → 0 as document length grows and query scope narrows.

(iv)

In expectation across the query distribution, I(q ; c) ≈ I(q ; d)relevant when d contains c — both recover the same relevant signal. But tokens(c) ≪ tokens(d). Therefore:

𝔼[Eff(q, c)] = 𝔼[I(q ; c)] / Tc > 𝔼[I(q ; d)relevant] / Td = 𝔼[Eff(q, d)] □
Formal Rebuttals to Partial Solutions

Three partial solutions are commonly proposed as alternatives to the concept graph architecture. Each is absorbed into the existing formalism.

Objection 1 — Document chunking A chunk is a document with smaller tokens(n). Define Tch = average chunk size where Tc < Tch < Td. The efficiency ratio Eff(q, chunk) = I(q; chunk) / Tch improves over Eff(q, document). However, chunk boundaries are determined by token count or sentence windows — not by concept boundaries. Therefore P(k | chunk) remains diffuse: a chunk may contain partial concept explanation blended with irrelevant content. The dominance result Eff(q, c) > Eff(q, chunk) holds because tokens(c) is bounded by concept scope while I(q; chunk) < I(q; c) due to dilution. The efficiency gap narrows but does not close. Chunking is a document-side improvement that does not address the fundamental granularity mismatch.
Objection 2 — Solr / metadata index A metadata index resolves the navigation problem — get to the right document faster — but does not address coverage-per-token efficiency. Once the document is retrieved, the full token cost Td is still paid. Formally: the metadata index reduces retrieval latency but does not change I(q; d) or tokens(d). Eff(q, d) is unchanged. The index is a retrieval filter, not a knowledge representation.
Objection 3 — Enriched document metadata with concept/instance references If documents are tagged with canonical concept identifiers, Coverage(q, G) improves because documents can be retrieved by concept membership rather than keyword. This is the best document-centric approach. However, set membership {c₁, c₂, c₃} encodes presence but not structure. Two failures follow: (1) directionality is lost — prerequisite-of relationships between c₁ and c₂ are not representable in a tag set; the system cannot traverse dependency chains or rank retrieval order by conceptual dependency. (2) edge information content is lost — the typed predicate τ in an edge (c₁, τ, c₂) carries conditional information I(q; c₂ | c₁, τ) − I(q; c₂) that is strictly greater than zero for well-typed graphs. Tag-based approaches discard this lift entirely. Enriched metadata improves recall; it does not recover the information content of typed directed edges.
Extension: Instance Layer Efficiency

The concept layer addresses queries over knowledge units — what things are and how they relate. A second, separable layer addresses queries over instances — specific entities, their properties, and their relationships. The efficiency argument extends structurally intact.

Define an instance query as one that induces a distribution I(q) over specific entities and properties: not "what is NAV methodology" but "what was Carlyle CCP V's reported NAV at Q3 2024." Define an instance node as a typed entity with canonical identity and encoded properties.

Instance layer efficiency
Eff(q, instance node) = I(q ; instance node) / tokens(instance node)

// By the same argument: I(q ; instance node) ≈ I(q ; d)relevant when d contains the entity
// But tokens(instance node) << tokens(d) — a structured fact is far smaller than the document containing it
// Therefore Eff(q, instance node) > Eff(q, d) by the same mechanism as the concept layer

The entity resolution problem is the instance-layer analog of semantic bridging at the concept layer. "JPMorgan," "JPMorgan AM," "JPMorgan Asset Management," and a specific JPMorgan line of business are one entity or several, depending on canonical resolution. Without canonical instance nodes, queries over entity relationships return incomplete and inconsistent results — the same entity contributes to C(q) under multiple surface forms with no structural link.

Two-layer coverage The full coverage of query q decomposes into conceptual coverage Coverage(q, Gconcept) and instance coverage Coverage(q, Ginstance). Both are required for queries that have both conceptual dependencies (what is NAV methodology?) and instance dependencies (what was fund X's NAV?). The layers are separable — each delivers independent efficiency gains — and composable — together they cover the full query space without redundancy.

open: instance construction cost Instance graphs have higher construction cost per node (entity extraction, canonical resolution, deduplication, incremental refresh) and higher volatility than concept graphs. The amortization crossover N* differs. Full derivation is an open task.

dependency chains Pre-encoded concept dependencies reduce expected inference steps E[IG(q)] versus E[I0(q)]. Savings compound with chain depth.

instantiation depth H(c | d) — residual uncertainty about concept c after reading document d — is positive even when d contains c. A concept node drives H(c | node) → 0 by construction.

C(q) approximation The semantic bridging step is separable. Tractable estimation strategies in priority order: (1) LLM decomposition — day-one available, highest cost; (2) embedding similarity — cheap once infrastructure exists; (3) historical traversal patterns — near-zero cost as query logs accumulate. Cost of C(q) estimation decreases monotonically as the system matures.
