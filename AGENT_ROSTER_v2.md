## Purpose of This Document

This document is the authoritative specification for the PKIS agent system. It supersedes
the agent descriptions in `PKIS_Wiki_Implementation_Plan.md` (v1.0) where they conflict.

Read this document in full before modifying any agent `.md` file, adding a new agent,
or implementing any new artifact (inbox, proposals directory, usage store). The
implementation order in § Build Order is mandatory — do not skip ahead.

Existing agent files (`LIBRARIAN.md` v3.0, `SYNTHESIZER.md` v2.0, `MAINTENANCE.md` v2.0,
`HYGIENIST.md` v1.0, `SCHEMA.md` v3.0) remain authoritative for their own procedure
details. This document specifies scope changes, new responsibilities, new artifacts,
and the relationships between agents. Where this document and an existing agent file
conflict, this document wins — update the agent file to match.

## System Overview

The PKIS agent system consists of six agents operating on a shared wiki graph backed
by markdown files in git. Agents are Claude Code skills invoked manually or on schedule.
They share a unified review surface (`wiki/inbox.md`) where all items requiring human
attention are surfaced.

### Agent Map

| Agent | Primary Role | Schedulable? | Writes to Graph? | Human Approval Required? |
|---|---|---|---|---|
| Hygienist | Schema conformance and conventions enforcement | Yes (periodic sweep) | No — report only | No |
| Librarian | Source ingest and reading graph interrogation | No — requires input | Yes | No (ingest is autonomous) |
| Synthesizer | Node deepening, connection discovery, inbox integration | Yes (proposal pass) | Yes — after approval | Yes |
| Maintenance | Structural graph health checks | Yes (fortnightly) | Minimal auto-fix only | No |
| Architect | Project self-knowledge, CONTEXT.md, architecture drift | Yes (post-structural-change or weekly) | CONTEXT.md and architecture doc only | No |
| Comptroller | API cost tracking and budget reporting | Yes (daily/weekly) | Usage store only | No |

### Shared Artifacts

All agents read from and write to a consistent set of shared artifacts:

```
wiki/
  inbox.md              # unified review queue — all agents append; human drains
  proposals/            # Synthesizer async proposal documents
    synthesizer-YYYY-MM-DD.md
  maintenance_report_YYYY-MM-DD.md
  architect_report_YYYY-MM-DD.md
  log.md                # append-only operation log

pkis-root/
  CONTEXT.md            # session primer — generated by Architect, never hand-edited
  AGENT_ROSTER_v2.md    # this document

hetzner-vps/
  /var/pkis/usage/
    usage.sqlite        # Comptroller usage store
    comptroller_report_YYYY-MM-DD.md
```

## Schema Addition: Discovery Stub Node Type

Before implementing any agent changes, add `discovery_stub` as a new node type to
`SCHEMA.md`. This is a prerequisite for the Librarian and unified inbox changes below.

### What It Is

A `discovery_stub` is a lightweight placeholder for a source encountered during
discovery — seen, potentially relevant, not yet worth full ingest. It carries enough
information to surface the source when its primary concept becomes relevant, without
adding hub-degree nodes to the live graph.

### Structural Constraints (enforced by convention in LIBRARIAN.md)

- **Maximum two concept links** — links only to the one or two most directly relevant
  concept or technique nodes. No connections to other source nodes. No connections to
  framework, result, or principle nodes.
- **No Reading Path section** — discovery stubs do not carry a Reading Path. They are
  not queued for reading; they are parked for potential future ingest.
- **No Key Extractions** — content is not yet extracted. The rationale field is the
  only substantive content.
- **Promotion path** — a discovery stub may be promoted to a full source node via
  the Librarian's ingest workflow. Promotion is a deliberate human decision, not automatic.

### Frontmatter Template

Add to `SCHEMA.md` § Frontmatter Templates:

```yaml
---
id: "pkis:discovery-stub:{slug}"   # assigned at creation; never changed
aliases: []
title: ""
authors: ""
year:
type: paper | book | book-chapter | article | talk | unknown
domain: []
tags: []
source_url: ""          # URL if available; null otherwise
knowledge_type: discovery-stub
status: parked          # always parked until promoted — do not use unread/in-progress/read
date_added: YYYY-MM-DD
rationale: ""           # why this source might be relevant — one to three sentences
primary_concepts: []    # maximum two wikilinks — the concepts that make this potentially relevant
promoted: false         # set to true when promoted to full source node; never delete the stub
promoted_to: ""         # wikilink to the full source node after promotion; null until promoted
---
```

### Folder

Discovery stubs live in `wiki/discovery/`. Add this folder to the repo structure.

### IRI Convention

`pkis:discovery-stub:{slug}` — add to `SCHEMA.md` § IRI Identity table:

| Folder | `knowledge_type` value | IRI prefix |
|---|---|---|
| `wiki/discovery/` | `discovery-stub` | `pkis:discovery-stub:` |

### Index Convention

Add a `## Discovery` section to `wiki/index.md`:

```markdown
## Discovery
- [[slug]] — rationale summary (primary concept) (YYYY-MM-DD) [promoted: yes/no]
```

### Existing Discovery Inbox Items

After implementing the discovery stub schema, recast existing items in the discovery
inbox as `discovery_stub` nodes. Each item gets a stub file in `wiki/discovery/` with
its rationale preserved. The discovery inbox view in the PWA should read from
`wiki/discovery/` filtered to `promoted: false`.

### 1. Hygienist (Expanded Scope)

**Current version:** 1.0 — Drive PDF filename normalization only  
**New version:** 2.0 — Full schema conformance and conventions enforcement

#### Scope Change

The Hygienist's mandate expands from Drive filename normalization to ownership of
all conventions enforcement across the entire PKIS project. The Drive filename work
remains but is one check among many.

The core question the Hygienist answers: *does every artifact in this project conform
to SCHEMA.md?*

#### Trigger Conditions

- Before a Librarian ingest session (pre-flight)
- After any structural change to SCHEMA.md
- As a periodic sweep (weekly recommended)
- On demand: `Hygienist, run conformance check [scope]`

Scope may be: `full`, `wiki-only`, `drive-only`, `discovery-only`, or a specific
node type: `concepts`, `techniques`, etc.

#### Checks (v2.0)

**Check 1: Drive Filename Normalization** (existing — unchanged)
Normalize PDF filenames in `PKIS/sources/papers/` and `PKIS/sources/books/`.
See existing HYGIENIST.md for procedure.

**Check 2: IRI Completeness**
Every node across all wiki folders must have `id` as its first frontmatter field
in `pkis:{type}:{slug}` format. Flag any node missing `id`. Flag any node where
`id` does not match the file's slug or folder.

**Check 3: Slug Uniqueness**
Scan all 12 wiki subdirectories (including `wiki/discovery/`). Flag any slug that
appears in more than one folder.

**Check 4: Alias Population**
Flag concept, technique, result, framework, problem, and principle nodes where
`aliases: []` but the node has a commonly known abbreviation or synonym (detectable
from title or content). Surface these as suggestions, not auto-fixes.

**Check 5: Frontmatter Completeness**
For each node type, check that all required frontmatter fields are present and
non-empty. Required fields by type are defined in SCHEMA.md § Frontmatter Templates.
Flag missing or empty required fields.

**Check 6: Reading Path Edge Type Labels**
Scan all `## Reading Path` sections. Every entry must carry either `[orientation]`
or `[integration]` label. Flag entries missing the label.

**Check 7: Discovery Stub Constraint Enforcement**
For all nodes in `wiki/discovery/`: flag any stub with more than two entries in
`primary_concepts`, any stub with a `## Reading Path` section, any stub with a
`## Key Extractions` section, and any stub with `promoted: true` that has no
`promoted_to` value.

**Check 8: Component Score Consistency**
Verify that `component_scores` keys in each node match the expected keys for its
`knowledge_type` per SCHEMA.md. Flag nodes where the key set is wrong or missing
entirely. Do not modify scores — flag only.

**Check 9: Cross-Agent Convention Consistency**
Read all agent `.md` files and SCHEMA.md. Flag any agent that references a node
type, folder, predicate, or field that does not exist in the current SCHEMA.md.
This is the lightweight version of the Architect's consistency check — the
Hygienist catches syntactic violations; the Architect catches semantic drift.

#### Output

Writes `wiki/hygiene_report_YYYY-MM-DD.md`. Auto-fixes: none (Hygienist v2.0 is
report-only except for the Drive rename procedure, which requires user approval
per existing procedure). Appends a summary line to `wiki/inbox.md` under
`## Conformance` swim lane.

#### Execution Environment

Claude Code. No git commit — reports only.

### 2. Librarian (Minor Changes)

**Current version:** 3.0 — unchanged in procedure  
**Changes in v2.0 roster:** inbox integration, discovery stub creation path

#### New: Discovery Stub Creation Path

Add to LIBRARIAN.md § Ingest Workflow a new pre-ingest decision:

Before beginning a full ingest, the Librarian may be invoked to create a discovery
stub instead: `Librarian, park [source]`

This creates a `discovery_stub` node in `wiki/discovery/` with:
- Title and authors from metadata (fetch if URL, read from Drive if file)
- Rationale provided by the user in the invocation or prompted for if absent
- Up to two primary concept links — Librarian proposes; user confirms

The stub is committed and appended to `wiki/inbox.md` under `## Discovery` as a
parked item. No queue entry is created.

#### New: Inbox Append at Session End

At the end of every Librarian ingest session, before committing, append to
`wiki/inbox.md` under the appropriate swim lane:

- Any source where classification confidence was low → `## Awaiting Classification`
- Any discovery stub created this session → `## Discovery`
- Any source where Drive ID could not be confirmed → `## Conformance`

#### Execution Environment

Claude Code. Git commit at session end (unchanged).

### 3. Synthesizer (Significant Changes)

**Current version:** 2.0  
**Changes in v2.0 roster:** async proposal mode, inbox integration, schedulable periodic pass

#### New: Async Proposal Mode

The Synthesizer gains a fifth operating mode alongside the existing four.

**Mode 5: Periodic Proposal Pass**

Triggered: `Synthesizer, run proposal pass` or on schedule (see below).

This mode runs without a synchronous conversation and without requiring immediate
human approval. It produces a structured proposal document and exits.

Procedure:
1. Read `wiki/inbox.md` § `## Structural Gaps` for Maintenance-flagged items
2. Run connection discovery across the two domains with the lowest cross-domain
   edge counts (from the most recent Maintenance report)
3. Identify the top 5 stub graduation candidates (stubs with 3+ sources, per
   Maintenance Check 9)
4. For each graduation candidate, draft a deepening proposal using the appropriate
   template
5. Write all proposals to `wiki/proposals/synthesizer-YYYY-MM-DD.md`
6. Append a summary to `wiki/inbox.md` under `## Proposals` swim lane:
   `Synthesizer proposal pass YYYY-MM-DD — N proposals ready for review`
7. Commit: `[synthesizer] proposal pass YYYY-MM-DD`

The proposal document format:

```markdown
# Synthesizer Proposal Pass — YYYY-MM-DD

## Connection Proposals
[ranked list per existing Mode 2 format]

## Deepening Proposals
### [[node-slug]] (technique)
[full draft per appropriate template]
[APPROVE / REJECT / DEFER — human marks inline]

## Structural Gap Responses
[responses to Maintenance-flagged gaps]
```

**Execution of approved proposals:**

`Synthesizer, execute proposals YYYY-MM-DD`

Reads the proposal document, finds all items marked APPROVE, executes them in
order, commits. Items marked REJECT are logged to `wiki/log.md`. Items marked
DEFER remain in the proposal document for the next pass.

#### Schedulable

The periodic proposal pass can run unattended. Recommended cadence: weekly.
Trigger via cron on Hetzner calling Claude Code CLI, or manually.

Approval execution is always manual — the human marks the proposal document,
then invokes execution.

#### Execution Environment

Claude Code. Git commit on proposal generation and on execution.

### 5. Architect (New Agent)

**Version:** 1.0  
**File to create:** `ARCHITECT.md`

#### Role

The Architect owns PKIS project self-knowledge. It is the only agent whose job
is to look at the system itself rather than the content of the graph. Three
responsibilities:

1. **Schema and agent consistency** — do all agent `.md` files use node types,
   predicates, fields, and folder names that exist in the current SCHEMA.md?
2. **Architecture document maintenance** — does the architecture document reflect
   the system as it actually exists?
3. **CONTEXT.md generation** — produce a current, accurate session primer from
   ground truth.

#### Trigger Conditions

- After any session that changes SCHEMA.md, adds or modifies an agent `.md` file,
  or changes MCP server behavior
- Weekly scheduled run as a backstop
- On demand: `Architect, run`

#### Tools Required

- Read access to entire pkis repo
- Read access to `AGENT_ROSTER_v2.md` (this document)
- Write access to `CONTEXT.md` and the architecture document only
- No MCP server access required — reads files only

#### Procedure

**Step 1: Read ground truth**
Read: `SCHEMA.md`, all agent `.md` files, `AGENT_ROSTER_v2.md`, the current
`CONTEXT.md`, the architecture document, and the MCP server's tool list (via
`get_write_schema` if available, otherwise from MCP server source files).

**Step 2: Consistency check**
For each agent `.md` file, verify:
- All referenced node types exist in SCHEMA.md
- All referenced predicates exist in SCHEMA.md § Relationship Predicates
- All referenced folders exist in the repo structure
- All referenced tools exist in the agent's declared tool set
- Write permissions in the agent file match actual folder existence

Flag all violations. These are cross-agent conformance issues distinct from
per-node conformance (which is Hygienist's domain).

**Step 3: Architecture drift check**
Compare the architecture document against:
- Actual agent files (are all agents documented? are capabilities accurate?)
- Actual repo structure (do all documented folders exist?)
- SCHEMA.md (are documented node types and predicates current?)

Flag items where the architecture document has drifted from reality. Propose
updates. Do not auto-update — present proposed changes for confirmation.

**Step 4: Generate CONTEXT.md**
Write a fresh `CONTEXT.md` from the ground truth read in Step 1. See
CONTEXT.md Specification below.

**Step 5: Write Architect report**
Write `wiki/architect_report_YYYY-MM-DD.md` with:
- Consistency violations found (and whether they were auto-fixable)
- Architecture drift items with proposed updates
- CONTEXT.md generation timestamp and summary of what changed

Append a one-line summary to `wiki/inbox.md` under `## Conformance`:
`Architect report YYYY-MM-DD — N consistency violations, M drift items`

#### CONTEXT.md Specification

`CONTEXT.md` lives at the repo root. It is generated by the Architect and
**never hand-edited**. It is the first file any Claude session should read
before engaging with PKIS.

Contents (two pages maximum):

### 6. Comptroller (New Agent)

**Version:** 1.0  
**File to create:** `COMPTROLLER.md`

#### Role

Track the total budgetary cost of all Claude API usage across all call sites.
Produce breakdowns by origin, model, project, and time window. Flag threshold
crossings. This agent owns the usage store and all budget reporting.

#### Cost Model

Two cost categories:

**Fixed:** Claude.ai Max Plan subscription. Record as a config constant in the
usage store: monthly amount, billing cycle start date. The Comptroller includes
this in monthly budget rollups as a line item but does not attempt to track it
dynamically.

**Variable:** API calls from all instrumented call sites. Cost is computed from
published Anthropic pricing at the time of each call: `(input_tokens × input_rate)
+ (output_tokens × output_rate) + (cache_read_tokens × cache_rate)`. Pricing
constants are stored in the usage store config table and updated when Anthropic
changes rates.

#### Data Architecture

**Usage store:** SQLite at `/var/pkis/usage/usage.sqlite` on Hetzner VPS.

Three tables:

```sql
-- Config: pricing constants and fixed costs
CREATE TABLE config (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Usage events: one row per API call
CREATE TABLE usage_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    emitted_at TIMESTAMP NOT NULL,
    origin TEXT NOT NULL,        -- 'pkis-mcp', 'claude-code', 'iks-pipeline', 'manual'
    project TEXT,                -- 'pkis', 'iks', 'ars', null if unknown
    model TEXT NOT NULL,         -- 'claude-sonnet-4-6', etc.
    input_tokens INTEGER NOT NULL,
    output_tokens INTEGER NOT NULL,
    cache_read_tokens INTEGER DEFAULT 0,
    cache_write_tokens INTEGER DEFAULT 0,
    computed_cost_usd REAL NOT NULL,
    attributes TEXT              -- JSON blob for extensible fields; schema evolves here
);

-- Budget periods: monthly rollups
CREATE TABLE budget_periods (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    period_start DATE NOT NULL,
    period_end DATE NOT NULL,
    fixed_cost_usd REAL NOT NULL,
    variable_cost_usd REAL NOT NULL,
    total_cost_usd REAL NOT NULL,
    generated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**The `attributes` JSON column** is the schema evolution surface. When a new
attribution dimension is needed (e.g., agent name, hypothesis cluster, IKS
pipeline job ID), add it to `attributes` without a migration. The Comptroller
reads it with known keys and ignores unknown ones. This is the open attributes
bag pattern — fixed envelope, flexible payload.

#### Instrumentation Points

**PKIS MCP Server (primary, highest priority)**
The MCP server is the most tractable instrumentation point. Every tool handler
that makes a Claude API call receives the response's `usage` metadata. Add a
`log_usage` function called at the end of each such handler:

```python
def log_usage(response, origin="pkis-mcp", project="pkis", attributes=None):
    usage = response.usage
    cost = compute_cost(usage.input_tokens, usage.output_tokens,
                        getattr(usage, 'cache_read_input_tokens', 0),
                        getattr(usage, 'cache_creation_input_tokens', 0))
    db.execute("""
        INSERT INTO usage_events
        (emitted_at, origin, project, model, input_tokens, output_tokens,
         cache_read_tokens, cache_write_tokens, computed_cost_usd, attributes)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
    """, (datetime.utcnow(), origin, project, response.model,
          usage.input_tokens, usage.output_tokens,
          getattr(usage, 'cache_read_input_tokens', 0),
          getattr(usage, 'cache_creation_input_tokens', 0),
          cost, json.dumps(attributes or {})))
```

**Claude Code (secondary, partial)**
Claude Code makes API calls through its own internals. Check for a local usage
file at `~/.claude/usage.json` or equivalent on the machine where Claude Code
runs. If present, the Comptroller ingests new entries from this file on each
run, deduplicating by session ID. If absent, Claude Code usage is tracked only
as an estimated line item in the budget model based on observed session patterns.

**Future call sites (IKS pipeline, ARS)**
When IKS pipelines are built, they wrap the Anthropic SDK client in the same
`log_usage` pattern. The `origin` field distinguishes them: `'iks-pipeline'`,
`'ars-session'`, etc. No architecture change required — the usage store already
accommodates arbitrary origins.

#### Pricing Config Initialization

On first run, seed the config table:

```sql
INSERT INTO config VALUES ('claude_sonnet_input_per_mtok', '3.00', CURRENT_TIMESTAMP);
INSERT INTO config VALUES ('claude_sonnet_output_per_mtok', '15.00', CURRENT_TIMESTAMP);
INSERT INTO config VALUES ('claude_sonnet_cache_read_per_mtok', '0.30', CURRENT_TIMESTAMP);
INSERT INTO config VALUES ('claude_sonnet_cache_write_per_mtok', '3.75', CURRENT_TIMESTAMP);
INSERT INTO config VALUES ('max_plan_monthly_usd', '100.00', CURRENT_TIMESTAMP);
INSERT INTO config VALUES ('billing_cycle_start_day', '1', CURRENT_TIMESTAMP);
```

Update pricing config manually when Anthropic changes rates. The Comptroller
reads these values at report time — historical events retain the cost computed
at emission time and are not retroactively repriced.

#### Trigger Conditions

- Daily: summary of previous day's variable API costs by origin
- Weekly: full breakdown by origin, project, model, with week-over-week comparison
- Monthly: budget close — fixed + variable, by period
- On demand: `Comptroller, report [daily|weekly|monthly|ytd]`
- On threshold crossing: when weekly variable cost exceeds a configurable threshold
  (default: 50% of monthly budget headroom), write an alert to `wiki/inbox.md`
  under `## Budget`

#### Report Format

Write to `/var/pkis/usage/comptroller_report_YYYY-MM-DD.md`:

```markdown
# Comptroller Report — YYYY-MM-DD [daily|weekly|monthly]

## Budget Summary
Period: YYYY-MM-DD to YYYY-MM-DD
Fixed cost (Max Plan): $XX.XX
Variable cost (API): $XX.XX
Total: $XX.XX
Remaining headroom: $XX.XX

## Variable Cost Breakdown

### By Origin
| Origin | Calls | Input Tokens | Output Tokens | Cost |
|---|---|---|---|---|
| pkis-mcp | N | N | N | $X.XX |
| claude-code | N | N | N | $X.XX (estimated) |

### By Model
| Model | Calls | Cost |
|---|---|---|

### By Project
| Project | Cost | % of Variable |
|---|---|---|

### By Day (weekly/monthly reports)
[sparkline or table]

## Alerts
[threshold crossings, anomalies]
```

#### Execution Environment

Runs on Hetzner VPS. The Comptroller is a Python script, not a Claude Code
agent — it reads from the SQLite store and produces reports without an LLM call.
It is the one agent in the roster that does not require Claude. Schedulable via
cron. The `COMPTROLLER.md` file describes the script's behavior and configuration,
not an LLM agent procedure.

**Exception:** the `Comptroller, report` on-demand invocation can be triggered
from a Claude Code session, which calls the script and returns the output.

## Unified Review Queue: wiki/inbox.md

`wiki/inbox.md` is the single surface where all items requiring human attention
are collected. All agents append to it; only the human removes items (by checking
them off or deleting lines).

### Structure

```markdown
# PKIS Inbox
Last updated: YYYY-MM-DDTHH:MM:SSZ

## Discovery
Items parked for potential future ingest. Review when a primary concept becomes active.
- [ ] [[slug]] — rationale (primary concept) (date parked) [Librarian]

## Proposals
Synthesizer proposals ready for review and approval.
- [ ] Synthesizer proposal pass YYYY-MM-DD — N proposals (wiki/proposals/synthesizer-YYYY-MM-DD.md)

## Staged
Bridge notes and staged nodes awaiting commit or discard.
- [ ] [[staged-id]] — title (date staged) [MCP]

## Structural Gaps
Maintenance-flagged gaps for Synthesizer attention.
- [ ] Problem without techniques: [[problem-slug]] (date flagged) [Maintenance]
- [ ] Isolated domain: domain-name — N cross-domain edges (date flagged) [Maintenance]

## Awaiting Classification
Sources with low-confidence knowledge object classifications pending human decision.
- [ ] [[source-slug]] § object-name — candidate types: technique or principle (date flagged) [Librarian]

## Conformance
Schema violations and convention issues.
- [ ] [[node-slug]] — missing IRI (date flagged) [Hygienist]
- [ ] Architect report YYYY-MM-DD — N violations (date)

## Budget
Cost alerts from Comptroller.
- [ ] Weekly variable cost $XX.XX — exceeds threshold (date)
```

### Rules

- Agents append only — never remove or reorder existing items
- Human checks off items when resolved: `- [x]` — or deletes the line
- The PWA inbox view reads `wiki/inbox.md` and renders each swim lane as a section
- Checked-off items older than 30 days are pruned by Maintenance (Check 5 extension)
